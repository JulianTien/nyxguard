## 2 项目介绍

### 2.1 项目说明

- **项目名称**：NyxGuard — 女生夜归安全守护APP
- **任务提出者**：NyxGuard 项目组
- **开发者**：NyxGuard 项目组（3-5名学生）
- **用户群**：女性用户（核心）、守护者（家人/朋友）

### 2.2 项目背景

女性夜间出行安全是一个普遍且真实的社会痛点。现有解决方案存在明显不足：

| 痛点场景     | 现有方案         | 不足                                   |
| ------------ | ---------------- | -------------------------------------- |
| 走夜路害怕   | 发微信报平安     | 出发到到达之间是"信息黑箱"，出事才知道 |
| 打车不安心   | 分享车牌号给朋友 | 不知道车往哪开，无法实时预警           |
| 想找借口脱身 | 假装打电话       | 不够逼真，容易被识破                   |
| 感到孤独害怕 | 独自忍受         | 缺乏情感陪伴与安全感                   |

NyxGuard 旨在通过**实时位置守护 + AI情感陪伴 + 智能预警**三位一体的方案，填补"出发到到达"之间的安全监护空白。本项目同时作为 Technovation Girls 国际比赛参赛作品。

### 2.3 项目目标

1. 开发一款可运行的 Android APP，包含步行模式与乘车模式双核心功能
2. 部署稳定运行的 FastAPI 后端服务，并采用独立服务部署方式
3. 集成 AI 大模型实现情境感知的关怀对话能力
4. 满足 Technovation Girls 技术提交要求（APK + 演示视频 + 源码仓库）
5. 在 24 课时（12周）内完成全流程交付

### 2.4 技术与部署口径

- 移动端主应用为 Android APP
- 服务端主线架构为 FastAPI + SQLAlchemy + JWT
- 生产数据库为 Neon Postgres
- 本地开发、演示和离线调试允许使用 SQLite fallback
- 生产部署采用 FastAPI 独立部署；Vercel 用于承载项目官网、静态内容、文档站或演示页

### 2.4 项目用户

| 用户类别            | 说明                                  | 使用频率          |
| ------------------- | ------------------------------------- | ----------------- |
| 普通用户（女生）    | 核心用户，夜间出行时使用步行/乘车模式 | 高频（每周2-5次） |
| 守护者（家人/朋友） | 接收行程通知，实时查看用户位置        | 被动触发          |

## 3 需求说明

### 3.1 整体需求

NyxGuard 系统需满足以下整体需求：

1. **实时守护**：用户出行期间，系统持续追踪位置并上报后端，守护者可实时查看
2. **智能预警**：系统自动检测超时、路线偏离等异常情况，主动触发预警通知
3. **AI陪伴**：通过大语言模型提供情境感知的关怀对话，缓解用户夜间出行的恐惧感
4. **紧急求助**：提供一键SOS功能，触发后自动向守护者发送位置、录音等关键信息
5. **隐蔽脱身**：模拟来电功能帮助用户在尴尬或危险场景中自然脱身
6. **主动威慑**：提供警报闪光功能，通过高频屏幕闪烁与震动快速引起周围注意
7. **离线可用**：核心定位与本地缓存功能在断网时仍可工作，恢复网络后自动补传数据

### 3.2 功能需求

#### 3.2.1 需求编号规则

需求编号格式：**SW + 模块代码 + 序号**

- `SW`：项目名称缩写（NyxGuard）
- 模块代码：两位字母，标识功能模块
- 序号：三位数字，标识模块内功能点

示例：`NGBX001` 表示 NyxGuard 步行模式模块的第1个功能点。

#### 3.2.2 总体模块划分

根据业务功能划分，NyxGuard 系统共分为以下 8 个功能模块：

| 模块代码 | 模块名称   | 优先级 | 说明                       |
| ---- | ------ | --- | ------------------------ |
| YH   | 用户管理   | P0  | 注册、登录、个人信息、守护者管理         |
| BX   | 步行模式   | P0  | 行程设置、实时追踪、超时检测、偏离检测、到达确认 |
| CC   | 乘车模式   | P0  | 车辆信息录入、位置追踪、偏离预警、一键SOS   |
| AI   | AI关怀对话 | P0  | 主动问候、情绪识别、异常询问、关怀回复      |
| MN   | 模拟来电   | P1  | 自定义来电界面、预录音频播放、延时触发      |
| BJ   | 警报闪光   | P1  | 全屏闪烁、震动威慑、长按关闭         |
| SOS  | 紧急求助   | P0  | 一键SOS、倒计时确认、录音、位置发送      |
| TZ   | 守护者通知  | P1  | 行程开始/超时/偏离/SOS事件推送       |

## 4 功能性需求

### 4.1 NGYH 用户管理模块

**模块名称**：用户管理

**模块简介**：提供用户注册、登录、个人信息管理及守护者管理功能，是系统的基础模块，所有其他功能模块依赖用户身份认证。

**模块功能列表**：

| 序号  | 一级功能   |         | 二级功能    |            |
| --- | ------ | ------- | ------- | ---------- |
|     | 功能名称   | 功能编号    | 功能名称    | 功能编号       |
| 1   | 用户注册   | NGYH001 |         |            |
| 2   | 用户登录   | NGYH002 |         |            |
| 3   | 个人信息管理 | NGYH003 | 查看个人信息  | NGYH003001 |
|     |        |         | 修改个人信息  | NGYH003002 |
| 4   | 守护者管理  | NGYH004 | 添加守护者   | NGYH004001 |
|     |        |         | 查看守护者列表 | NGYH004002 |
|     |        |         | 删除守护者   | NGYH004003 |

#### 4.1.1 用户注册 NGYH001

用户通过手机号或邮箱进行注册，注册时需填写以下信息：

- **昵称**：文本框输入，2-20个字符
- **手机号/邮箱**：文本框输入，作为登录账号，需格式校验
- **密码**：文本框输入，6-20位，需包含字母和数字
- **确认密码**：文本框输入，需与密码一致

注册成功后自动登录并跳转至首页。

#### 4.1.2 用户登录 NGYH002

用户通过手机号/邮箱 + 密码登录系统。

- 登录成功后获取 Token，后续请求携带 Token 进行身份认证
- 登录失败提示错误原因（账号不存在 / 密码错误）
- 支持"记住登录状态"，Token 有效期内免登录

#### 4.1.3 个人信息管理 NGYH003

##### 查看个人信息 NGYH003001

以页面形式展示当前用户的基本信息，包括：昵称、手机号/邮箱、头像、注册时间。

##### 修改个人信息 NGYH003002

支持修改以下字段：

- **昵称**：文本框输入
- **头像**：从相册选择或拍照上传
- **紧急联系电话**：文本框输入，用于SOS场景

#### 4.1.4 守护者管理 NGYH004

##### 添加守护者 NGYH004001

用户可添加守护者（家人/朋友），需填写：

- **守护者昵称**：文本框输入
- **守护者手机号**：文本框输入，用于接收通知
- **关系**：下拉选择（父母、朋友、伴侣、其他）

每个用户最多添加 5 名守护者。

##### 查看守护者列表 NGYH004002

以列表形式展示已添加的守护者信息，包括：昵称、手机号、关系、添加时间。

##### 删除守护者 NGYH004003

用户可删除已添加的守护者，删除前需二次确认。至少保留 1 名守护者方可使用步行/乘车模式。

### 4.2 NGBX 步行模式模块

**模块名称**：步行模式

**模块简介**：为用户夜间步行出行提供全程守护，包括行程设置、实时位置追踪、超时检测、路线偏离检测和到达确认等功能。核心理念："有人看着我走到家"。

**模块功能列表**：

| 序号 | 一级功能     |          | 二级功能         |            |
| ---- | ------------ | -------- | ---------------- | ---------- |
|      | 功能名称     | 功能编号 | 功能名称         | 功能编号   |
| 1    | 行程设置     | NGBX001  | 起终点设置       | NGBX001001 |
|      |              |          | 预计到达时间设置 | NGBX001002 |
|      |              |          | 选择守护者       | NGBX001003 |
| 2    | 路线规划     | NGBX002  |                  |            |
| 3    | 实时位置追踪 | NGBX003  |                  |            |
| 4    | 超时检测     | NGBX004  |                  |            |
| 5    | 路线偏离检测 | NGBX005  |                  |            |
| 6    | 到达确认     | NGBX006  |                  |            |

#### 4.2.1 行程设置 NGBX001

##### 起终点设置 NGBX001001

用户在步行模式页面设置行程的起点和终点：

- **起点**：默认使用当前GPS定位位置，也可手动输入地址或在地图上选点
- **终点**：通过文本框输入地址，支持高德地图POI搜索联想；也可在地图上点选位置
- 起终点设置完成后，地图上显示起终点标记

##### 预计到达时间设置 NGBX001002

用户设置预计到达终点的时间：

- 系统根据路线距离自动推荐预计用时（步行速度按5km/h估算）
- 用户可手动调整预计到达时间（滑块或时间选择器）
- 预计时间用于超时检测功能的判断基准

##### 选择守护者 NGBX001003

用户从已添加的守护者列表中选择本次行程的守护者：

- 以列表形式展示守护者昵称、关系
- 支持多选，至少选择1名守护者
- 选中的守护者将在行程开始时收到通知

#### 4.2.2 路线规划 NGBX002

用户确认起终点后，系统调用高德地图步行路线规划API，在地图上绘制推荐路线：

- 以折线形式在地图上绘制步行路线
- 显示预计步行距离和用时
- 路线数据缓存至本地，用于后续偏离检测的基准线
- 若路线规划失败（网络异常），提示用户检查网络并重试

#### 4.2.3 实时位置追踪 NGBX003

用户点击"开始行程"后，系统启动后台定位Service，持续追踪用户位置：

- 启动前台Service，通知栏常驻显示"NyxGuard 守护中"
- 定位频率：每10秒获取一次GPS位置
- 位置数据实时上报后端（每30秒批量上报一次）
- 地图上实时更新用户位置蓝点和已走过的轨迹
- 断网时位置数据缓存至Room本地数据库，恢复网络后自动补传

#### 4.2.4 超时检测 NGBX004

系统根据用户设置的预计到达时间，自动检测是否超时：

- 当前时间超过预计到达时间时，触发超时预警
- 超时后APP弹出提醒弹窗，用户可选择"我没事，延长时间"或"需要帮助"
- 若用户选择延长，可重新设置预计到达时间
- 若用户选择"需要帮助"，触发SOS流程
- 若超时后60秒内用户无响应，自动向守护者发送超时通知

#### 4.2.5 路线偏离检测 NGBX005

系统持续计算用户当前位置与规划路线的距离，检测是否偏离：

- 每次定位更新时，计算当前位置到规划路线折线的最短距离
- 偏离阈值：200米（步行模式）
- 偏离后APP弹出提醒："您已偏离规划路线，是否安全？"
- 用户可选择"我没事，换了条路"（更新路线）或"需要帮助"（触发SOS）
- 若偏离后60秒内用户无响应，自动向守护者发送偏离通知

#### 4.2.6 到达确认 NGBX006

系统检测用户是否到达终点：

- 当用户位置进入终点半径100米范围内时，判定为到达
- 弹出到达确认弹窗："您已到达目的地，确认安全到达？"
- 用户点击"确认到达"后，行程结束，停止定位Service
- 向守护者发送"已安全到达"通知
- 行程数据（轨迹、时长、距离）保存至后端

### 4.3 NGCC 乘车模式模块

**模块名称**：乘车模式

**模块简介**：为用户夜间乘车（网约车、出租车、顺风车等）提供安全守护，包括车辆信息录入、实时位置追踪、路线偏离预警和一键SOS功能。核心理念："有人知道我上了谁的车"。

**模块功能列表**：

| 序号 | 一级功能     |          | 二级功能     |            |
| ---- | ------------ | -------- | ------------ | ---------- |
|      | 功能名称     | 功能编号 | 功能名称     | 功能编号   |
| 1    | 车辆信息录入 | NGCC001  | 手动输入车牌 | NGCC001001 |
|      |              |          | OCR车牌识别  | NGCC001002 |
|      |              |          | 车辆描述     | NGCC001003 |
| 2    | 乘车行程设置 | NGCC002  |              |            |
| 3    | 乘车位置追踪 | NGCC003  |              |            |
| 4    | 乘车偏离预警 | NGCC004  |              |            |
| 5    | 一键SOS      | NGCC005  |              |            |

#### 4.3.1 车辆信息录入 NGCC001

##### 手动输入车牌 NGCC001001

用户手动输入车辆车牌号码：

- 文本框输入，支持中国大陆车牌格式校验（如：京A12345）
- 输入完成后显示车牌预览卡片
- 车牌信息随行程数据一并发送给守护者

##### OCR车牌识别 NGCC001002（P2 可选）

用户通过拍照自动识别车牌号码：

- 调用设备摄像头拍摄车牌照片
- 使用 ML Kit 文字识别提取车牌号
- 识别结果填入车牌输入框，用户可手动修正
- 若识别失败，提示用户手动输入（降级方案）

##### 车辆描述 NGCC001003

用户补充车辆外观描述信息：

- **车型**：下拉选择（轿车、SUV、面包车、其他）
- **颜色**：下拉选择（白色、黑色、银色、红色、蓝色、其他）
- **备注**：文本框输入，可填写平台名称（滴滴/高德/曹操等）或其他特征

#### 4.3.2 乘车行程设置 NGCC002

用户录入车辆信息后，设置乘车行程：

- **目的地**：文本框输入地址，支持高德地图POI搜索联想
- **选择守护者**：从守护者列表中选择本次行程的守护者（至少1名）
- 点击"开始乘车守护"后，车辆信息+目的地+实时位置一并发送给守护者
- 系统调用高德地图驾车路线规划API，获取推荐行车路线作为偏离检测基准

#### 4.3.3 乘车位置追踪 NGCC003

乘车守护开始后，系统启动后台定位Service，持续追踪用户位置：

- 启动前台Service，通知栏常驻显示"NyxGuard 乘车守护中"
- 定位频率：每5秒获取一次GPS位置（乘车速度较快，频率高于步行模式）
- 位置数据实时上报后端（每15秒批量上报一次）
- 地图上实时更新用户位置和行驶轨迹
- 断网时位置数据缓存至本地，恢复网络后自动补传

#### 4.3.4 乘车偏离预警 NGCC004

系统持续计算用户当前位置与规划行车路线的距离，检测车辆是否偏离：

- 每次定位更新时，计算当前位置到规划路线折线的最短距离
- 偏离阈值：500米（乘车模式，考虑司机可能选择不同路线）
- 偏离后APP弹出醒目预警："车辆已偏离预定路线！"
- 同时触发震动提醒（3次短震动）
- 用户可选择"司机换了条路，没事"（忽略本次偏离）或"我感到不安全"（触发SOS）
- 若偏离后30秒内用户无响应，自动向守护者发送偏离预警通知（含当前位置+车辆信息）

#### 4.3.5 一键SOS NGCC005

乘车模式下提供醒目的SOS按钮，用户感到危险时可一键触发紧急求助：

- 乘车追踪页面底部常驻红色SOS大按钮
- 点击SOS后进入5秒倒计时确认（防误触），倒计时期间可取消
- 倒计时结束后触发以下操作：
  - 自动开始录音（后台静默录音，持续至行程结束或手动停止）
  - 向所有已选守护者发送SOS通知（含当前位置、车辆信息、行驶轨迹）
  - 后端记录SOS事件，标记行程为"紧急状态"
  - 提高定位上报频率至每3秒一次
- SOS触发后，页面显示"紧急模式已启动"状态提示

#### 4.4.2 主动关怀触发 NGAI002

- **路线偏离**：AI发送"你好像走了不同的路，一切还好吗？"
- **长时间静止**：用户位置超过5分钟未变化时，AI发送"你停下来了，是在休息吗？还是需要帮助？"

#### 4.4.3 用户主动对话 NGAI003

- 消息通过后端 `/api/chat` 接口发送至LLM，获取AI回复
- AI回复基于System Prompt设定的"温暖守护者"人设，语气亲切、关怀
- AI具备行程上下文感知能力，能结合当前行程状态（步行中/乘车中/已到达）给出相关回复
- 网络异常时使用本地预设回复模板兜底，例如："网络不太好，但我一直在哦"

#### 4.4.4 情绪识别与回应 NGAI004

AI通过分析用户消息文本，识别用户情绪状态并给出针对性回应：

- **恐惧/害怕**：识别关键词（害怕、好黑、有人跟着等），回应安抚类消息并建议开启模拟来电或联系守护者
- **焦虑/紧张**：识别关键词（紧张、不安、心慌等），回应放松引导类消息
- **正常/开心**：正常关怀对话，轻松聊天陪伴
- 情绪识别通过LLM的System Prompt指令实现，无需额外模型
- 当识别到强烈负面情绪时，AI主动建议用户触发SOS或联系守护者

### 4.6 NGSOS 紧急求助模块

**模块名称**：紧急求助

**模块简介**：为用户提供一键SOS紧急求助功能，触发后自动向守护者发送位置信息、启动录音，并通知后端记录紧急事件。SOS功能贯穿步行模式和乘车模式，是用户最后的安全防线。

**模块功能列表**：

| 序号 | 一级功能      |          | 二级功能 |          |
| ---- | ------------- | -------- | -------- | -------- |
|      | 功能名称      | 功能编号 | 功能名称 | 功能编号 |
| 1    | SOS触发入口   | NGSOS001 |          |          |
| 2    | SOS倒计时确认 | NGSOS002 |          |          |
| 3    | 紧急录音      | NGSOS003 |          |          |

#### 4.6.1 SOS触发入口 NGSOS001

系统在多个位置提供SOS触发入口：

- **首页**：底部常驻红色SOS圆形按钮
- **步行追踪页**：页面底部SOS按钮
- **乘车追踪页**：页面底部SOS大按钮
- **通知栏**：行程进行中，通知栏快捷操作包含SOS按钮
- 所有入口触发后进入统一的SOS倒计时确认流程

#### 4.6.2 SOS倒计时确认 NGSOS002

SOS触发后进入倒计时确认流程，防止误触：

- 弹出全屏倒计时页面，显示5秒倒计时动画
- 倒计时期间页面显示"SOS即将发送，点击取消"
- 用户点击"取消"可中止SOS流程
- 倒计时结束后自动执行紧急求助操作（录音+通知+事件记录）
- 倒计时期间伴随震动反馈（每秒一次短震动）

#### 4.6.3 紧急录音 NGSOS003

SOS确认后自动启动后台录音：

- 使用设备麦克风进行环境录音
- 录音在后台静默进行，不显示明显的录音界面（保护用户安全）
- 录音文件保存至本地存储，格式为 AAC/M4A
- 录音持续至用户手动停止或行程结束
- 网络可用时，录音文件自动上传至后端服务器备份

#### 4.6.4 紧急通知发送 NGSOS004

SOS确认后自动向守护者发送紧急通知：

- 向所有已选守护者发送SOS推送通知
- 通知内容包含：用户昵称、当前GPS位置（经纬度+地址描述）、行程类型（步行/乘车）
- 乘车模式下额外包含：车牌号、车型、颜色等车辆信息
- 通知中附带地图链接，守护者点击可查看用户实时位置
- 若推送服务不可用，降级为后端日志记录

#### 4.6.5 SOS事件记录 NGSOS005

后端记录每次SOS事件的完整信息：

- 记录字段：用户ID、触发时间、触发位置（经纬度）、行程ID、行程类型
- 乘车模式额外记录：车牌号、车辆描述
- 标记关联行程状态为"紧急"
- SOS事件数据持久化存储，不可删除
- 后端提供SOS事件查询接口，供后续扩展使用

- 附带用户实时位置地图链接和行驶轨迹
- 通知优先级：紧急（突破静音模式，持续震动和响铃）
- 通知每隔2分钟重复发送，直到守护者确认已读或用户取消SOS

#### 4.7.3 到达通知 NGTZ003

用户确认安全到达后，自动向守护者发送到达通知：

- 通知内容："{用户昵称}已安全到达{终点地址}"
- 附带本次行程摘要：出发时间、到达时间、行程距离
- 通知优先级：普通

#### 4.7.4 通知模板管理 NGTZ004

后端维护通知消息模板，支持不同事件类型的通知内容定制：

- 每种事件类型（行程开始、超时、偏离、SOS、到达）对应一个通知模板
- 模板支持变量占位符替换（用户昵称、位置、车牌号等）
- 模板在后端配置，客户端无需硬编码通知内容
- 预留多语言支持扩展能力（当前版本仅支持中文）

## 5 非功能性需求

### 5.1 性能需求

| 序号 | 需求项         | 指标要求                           |
| ---- | -------------- | ---------------------------------- |
| 1    | 定位响应时间   | GPS定位首次获取 ≤ 5秒             |
| 2    | 位置上报延迟   | 客户端到后端 ≤ 3秒（网络正常时）  |
| 3    | AI对话响应时间 | 用户发送消息到收到AI回复 ≤ 5秒    |
| 4    | 页面加载时间   | 各页面首次加载 ≤ 2秒              |
| 5    | SOS触发延迟    | 倒计时结束到通知发出 ≤ 2秒        |
| 6    | 并发用户数     | 后端支持 ≤ 50 并发用户（MVP阶段） |

### 5.2 安全性需求

| 序号 | 需求项       | 说明                                             |
| ---- | ------------ | ------------------------------------------------ |
| 1    | 身份认证     | 使用 JWT Token 进行用户身份认证，Token 有效期7天 |
| 2    | 密码存储     | 用户密码使用 bcrypt 加盐哈希存储，禁止明文存储   |
| 3    | 传输加密     | 客户端与后端通信使用 HTTPS 协议                  |
| 4    | 位置数据保护 | 位置数据属于敏感信息，APP中需明确隐私政策说明    |
| 5    | API访问控制  | 所有业务接口需验证Token，防止未授权访问          |
| 6    | 录音数据保护 | SOS录音文件加密存储，仅授权用户和守护者可访问    |

### 5.3 兼容性需求

| 序号 | 需求项       | 说明                                                               |
| ---- | ------------ | ------------------------------------------------------------------ |
| 1    | Android版本  | 最低支持 Android 8.0（API 26），目标版本 Android 13（API 33）      |
| 2    | 屏幕适配     | 适配主流屏幕尺寸（5.0-6.7英寸），支持竖屏显示                      |
| 3    | 网络环境     | 支持 4G/5G 移动网络和 Wi-Fi 环境                                   |
| 4    | 后端运行环境 | Python 3.11+，采用 FastAPI 独立部署，兼容 Railway / Render / Fly.io 等平台 |
| 5    | 数据库兼容   | 本地开发与演示使用 SQLite fallback，生产环境使用 Neon Postgres，通过 SQLAlchemy ORM 屏蔽差异 |

### 5.4 可用性需求

| 序号 | 需求项   | 说明                                                               |
| ---- | -------- | ------------------------------------------------------------------ |
| 1    | 离线可用 | 核心定位功能在断网时仍可工作，位置数据本地缓存，恢复网络后自动补传 |
